# 非浪涌报警架构设计

本文定义 NTC 和 TI 开关量输入两类非浪涌报警的后端架构。浪涌报警、浪涌峰值事件和高速 ADC 采样不纳入本文。

## 设计目标

- 当前活跃报警掉电或复位后可恢复。
- 所有报警最终都形成可查询的历史日志。
- Modbus、UI、MSH 只读取快照或提交请求，不直接读写 EEPROM、Flash 或报警内部结构。
- 报警发生和结束要及时，但不能阻塞 Modbus 回调路径。
- EEPROM 写入只发生在报警状态边沿，不周期刷新持续时间。

## 数据分类

| 数据 | 存储位置 | 用途 | 写入时机 |
| --- | --- | --- | --- |
| 活跃报警 RAM 表 | RAM | 当前运行态查询 | 报警发生/结束 |
| 活跃报警恢复镜像 | AT24C128C EEPROM KVDB | 复位后恢复未闭合报警 | 报警发生/结束 |
| 报警历史事件日志 | AT32F403A 片内 Flash TSDB | 历史查询、客户追溯 | 报警发生写 start 事件，报警恢复写 clear 事件 |

## 模块职责

```text
data_service
  ↓ 周期提供 NTC / TI raw / RTC / sensor_valid 快照
alarm_service
  ↓ 维护活跃报警、生成报警 ID、判定发生/结束
param_service / alarm_active_store
  ↓ EEPROM KVDB 保存活跃报警恢复镜像
record_service
  ↓ 通过 record_wr 异步写 FlashDB TSDB 历史日志
UI / Modbus / MSH
  ↓ 只查询快照或调用服务接口
```

## 报警状态机

每个报警源维护独立状态：

```text
NORMAL
  ↓ 达到触发条件
ACTIVE
  ↓ 达到恢复条件
CLOSED
  ↓ 写历史记录后回到 NORMAL
```

报警发生流程：

1. `data_service` 周期读取 NTC、DI 和 RTC 快照。
2. `alarm_service_update()` 检测到报警从 `NORMAL` 进入 `ACTIVE`。
3. `alarm_service` 分配 `alarm_id`。
4. RAM 活跃报警表增加一项。
5. EEPROM KVDB 写入活跃报警镜像。
6. 生成一条 `alarm_start` 历史事件，由 `record_service` 投递给 `record_wr` 异步写入 Flash TSDB。
7. `data_service` 快照中的 `alarm_state` 置为 active。

报警发生时写 TSDB 的原因是让历史日志严格按照发生时间排序。该记录是完整的“报警发生事件”，不是等待后续回写的半成品记录。

报警结束流程：

1. `alarm_service_update()` 检测到报警从 `ACTIVE` 满足恢复条件。
2. 计算 `duration_ms`。
3. 生成一条 `alarm_clear` 历史事件，包含同一个 `alarm_id`、恢复时间、持续时间和报警原因。
4. `record_service` 将事件放入 RAM 环形缓存，并投递给 `record_wr` 异步写入 Flash TSDB。
5. EEPROM KVDB 删除或更新活跃报警镜像，移除该报警。
6. RAM 活跃报警表移除该报警。

不允许在报警恢复时回写或修改报警发生时已经写入的 TSDB 记录。FlashDB TSDB 按追加式事件日志使用，避免依赖裸 Flash 地址或触发查询路径扫描 Flash。

启动恢复流程：

1. `alarm_service_init()` 从 EEPROM KVDB 读取活跃报警镜像。
2. 校验 magic、version、CRC、数量和字段范围。
3. 对合法的未闭合报警恢复到 RAM 活跃报警表。
4. 恢复后的报警继续保持 active，`duration_ms` 由当前时间动态计算。
5. 如果 EEPROM 镜像损坏，只丢弃活跃恢复镜像，不清除历史 TSDB。

## 活跃报警 EEPROM 镜像

建议 KV 名称：

```text
active_alarm_image
```

建议镜像头：

| 字段 | 含义 |
| --- | --- |
| `magic` | 固定魔数，用于识别镜像 |
| `version` | 活跃报警镜像格式版本 |
| `count` | 当前活跃报警数量 |
| `next_alarm_id` | 下一个报警 ID，用于复位后继续递增 |
| `crc32` | 镜像校验 |

建议每条活跃报警字段：

| 字段 | 含义 |
| --- | --- |
| `alarm_id` | 报警 ID |
| `reason` | 报警原因 |
| `source_index` | 报警源编号，例如 TI1..TI5 |
| `start_timestamp_ms` | 发生 tick 时间 |
| `start_datetime` | 发生 RTC 时间 |
| `time_valid` | RTC 时间是否有效 |
| `ntc_temp_milli_c` | 发生时 NTC 温度 |
| `digital_input_state` | 发生时 DI raw 状态 |

容量估算：

- 当前最大活跃报警数量按 6 条规划：NTC 1 条 + TI1..TI5 共 5 条。
- 每个报警源最多只有一条当前活跃报警：TI1..TI5 各 1 条，NTC 1 条。
- NTC 高温和 NTC 低温共用同一个 NTC 活跃槽，同一时刻不允许同时存在两个 NTC 报警。
- 每条活跃报警按 32 到 48 字节估算，总镜像约 256 到 384 字节。
- AT24C128C EEPROM 总容量 16 KB，活跃报警镜像占用很小，可以接受。

## TI 命名和映射

业务层严格使用 `TI1..TI5`，底层采样层可以继续使用 `DI0..DI4` 或 bit0..bit4 表达原始 GPIO 状态。映射关系如下：

| 业务名 | 底层 bit | 当前 MCU 引脚 | 说明 |
| --- | --- | --- | --- |
| `TI1` | `DI0` / bit0 | `PA7` | 开关量输入 1 |
| `TI2` | `DI1` / bit1 | `PA6` | 开关量输入 2 |
| `TI3` | `DI2` / bit2 | `PA5` | 开关量输入 3 |
| `TI4` | `DI3` / bit3 | `PA4` | 开关量输入 4 |
| `TI5` | `DI4` / bit4 | `PA3` | 开关量输入 5 |

业务状态使用：

| 状态 | 业务含义 | 默认报警策略 |
| --- | --- | --- |
| `close` | 闭合 | 默认不报警 |
| `open` | 开路 | 默认触发报警 |

电平关系当前已确认：

| 条件 | MCU 输入电平 |
| --- | --- |
| 开路时 | 低电平 |
| 闭合时 | 高电平 |

默认配置应按 `open=低电平报警` 处理，即 `ti_alarm_open_level_mask = 0x00`。如果后续某一路现场接线相反，允许用户通过参数单独配置该路 open 电平。

## 参数规划

TI 有效极性和 NTC 阈值应进入参数服务，并通过 EEPROM KVDB 持久化。建议参数：

| 参数 | 含义 | 建议默认值 | 说明 |
| --- | --- | --- | --- |
| `ti_alarm_enable_mask` | TI 报警使能 bitmask | `0x00` | bit0..bit4 对应 TI1..TI5；产品默认关闭，需要用户显式启用 |
| `ti_alarm_open_level_mask` | TI open 状态电平 bitmask | `0x00` | bit 为 1 表示该路高电平为 open，0 表示低电平为 open；默认 open 为低电平 |
| `ntc_sensor_enable` | NTC 传感器使能 | `1` | 0 禁用，1 使能；传感器默认参与采样和业务状态 |
| `ntc_high_alarm_enable` | NTC 高温报警使能 | `0` | 产品默认关闭，需要用户显式启用 |
| `ntc_high_alarm_threshold_mc` | NTC 高温报警阈值 | `70000` | 单位 mC；Modbus 寄存器单位为 0.01 摄氏度 |
| `ntc_low_alarm_enable` | NTC 低温报警使能 | `0` | 产品默认关闭，需要用户显式启用 |
| `ntc_low_alarm_threshold_mc` | NTC 低温报警阈值 | `-20000` | 单位 mC；Modbus 寄存器单位为 0.01 摄氏度 |
| `ti_alarm_debounce_ms` | TI 报警消抖时间 | `50` | 防止输入抖动频繁写 EEPROM；报警发生和恢复边沿都需要稳定超过该时间 |
| `ntc_alarm_debounce_ms` | NTC 报警消抖时间 | `1000` | NTC 高温/低温报警发生和恢复都需要稳定超过该时间 |
| `surge_alarm_enable` | 浪涌报警使能 | `1` | 浪涌报警默认开启；浪涌详情参数另行设计 |

参数修改只能通过 `param_service` 或后续 `alarm_config_service`，Modbus/UI/MSH 不得直接改报警服务内部结构。

当前 NTC 不设置独立恢复阈值：高温报警在温度低于高温报警阈值后恢复，低温报警在温度高于低温报警阈值后恢复。发生和恢复都使用 `ntc_alarm_debounce_ms` 消抖。

参数默认策略：

- TI1..TI5 报警默认关闭，避免未接线或现场接线未配置时产生默认报警。
- NTC 高温/低温报警默认关闭，待阈值和安装条件确认后由用户或出厂配置启用。
- 浪涌报警默认开启；当前已有 `surge_alarm_enable` 参数，浪涌详情参数另行设计。
- 本固件尚未对外发行，不保留旧报警格式兼容；调试阶段已有参数镜像如不符合当前默认策略，可以通过恢复默认参数重新生成。

## 历史报警 TSDB 事件日志

报警历史使用 `RECORD_SERVICE_EVENT_ALARM`，字段见 `非浪涌报警记录格式.md`。

历史记录原则：

- 历史日志不放 EEPROM。
- 报警发生事件和报警恢复事件都必须进入片内 Flash TSDB。
- 报警发生事件按发生时间写入，报警恢复事件按恢复时间写入。
- 不合并 start/clear，不回写旧 TSDB 记录，不在前端查询路径扫描 TSDB 配对 `alarm_id`。
- 未闭合状态由 RAM 活跃报警表和 EEPROM KVDB 活跃报警镜像负责恢复。
- UI 和上位机展示时也按事件日志展示，例如“TI1 开路报警发生”和“TI1 开路报警恢复”。

## Modbus 和 UI 访问原则

- 当前活跃报警查询读取 RAM 快照，不直接读 EEPROM。
- 历史报警查询读取 `record_service` 提供的记录接口，不直接扫 TSDB。
- Modbus 回调不得执行 EEPROM KVDB 写入。
- Modbus 写清除报警、确认报警等操作时，应投递 `backend_request` 或后续 `alarm_request`。
- UI 只显示当前活跃报警和历史查询结果，不直接修改报警服务内部结构。

## 写入时序和磨损控制

- EEPROM 只在报警发生/结束边沿写，不按秒刷新。
- TI 输入报警发生和恢复都使用非阻塞消抖状态机，状态持续稳定超过 `ti_alarm_debounce_ms` 后才切换。
- NTC 不设置独立恢复回差或恢复阈值，按同一个高/低温阈值反向恢复。
- NTC 报警发生和恢复都使用非阻塞消抖状态机，状态持续稳定超过 `ntc_alarm_debounce_ms` 后才切换；实现只记录 pending 起始时间，不在线程中 sleep 等待。
- 后续应增加最小保持时间或写入合并策略，避免输入抖动造成频繁 KVDB 更新。
