# 网络通信收口测试记录

本文记录当前 Modbus TCP / W5500 网络通信在后端优先阶段已经收口的设计、寄存器和实测结论。本文只描述当前已验证范围，不代表最终产品手册。

## 当前设计结论

- Modbus RTU 和 Modbus TCP 共用同一套 Holding / Input 寄存器映射。
- 网络参数由 `param_service` 管理，EEPROM KVDB 持久化。
- 第一版推荐通过 RS485 / Modbus RTU 修改网络参数，避免通过 TCP 修改自身 IP 或端口后导致当前连接失联。
- W5500 当前只使用 `socket0` 作为 Modbus TCP server，第一版只支持 1 路 TCP client 并发连接。
- TCP 客户端连接后如果超过 5 秒不发送完整 Modbus TCP 请求，固件会主动关闭 socket0 并重新进入监听，避免单 socket 被异常客户端长期占用。
- DHCP 获取到的动态 IP 不写回 EEPROM，也不污染 Holding `51..63` 中的静态 / fallback 参数。
- DHCP 失败时回退到 Holding `51..63` 中的静态 / fallback 网络参数。

## Modbus 寄存器约定

### Holding 网络参数

| 地址 | 含义 |
| --- | --- |
| 50 | DHCP 使能，`0`=静态 IP，`1`=DHCP |
| 51..54 | 静态 / fallback IP 地址，每个寄存器保存一个字节 |
| 55..58 | 静态 / fallback 子网掩码 |
| 59..62 | 静态 / fallback 网关 |
| 63 | Modbus TCP 监听端口 |
| 64..69 | factory MAC，每个寄存器保存一个字节 |
| 100 | 异步命令，`6`=应用网络 / Modbus TCP 参数，`1`=保存参数 |
| 102 | 后台命令 busy |
| 103 | 最近后台命令结果，`0` 表示成功，负数按 int16 解释 |

### Input 网络运行状态

| 地址 | 含义 |
| --- | --- |
| 240 | flags：bit0=Modbus TCP 已运行，bit1=当前使用 DHCP 地址，bit2=正在应用网络参数，bit3=网线链路已连接，bit4=100M，bit5=全双工 |
| 241 | W5500 socket0 状态寄存器快照 |
| 242..245 | 当前实际运行 IP |
| 246..249 | 当前实际运行子网掩码 |
| 250..253 | 当前实际运行网关 |
| 254..257 | DHCP DNS，静态模式或 DHCP 未成功时为 0 |
| 258..259 | DHCP lease seconds，静态模式或 DHCP 未成功时为 0 |

## 已完成测试

### 静态网络启动和掉电恢复

测试条件：

- DHCP 关闭。
- 静态 IP 为 `192.168.1.223`。
- Modbus TCP 端口为 `502`。

测试结论：

- 掉电重新上电后，Holding `50..63` 读回 DHCP=`0`、IP=`192.168.1.223`、端口=`502`。
- TCP `192.168.1.223:502` 可连接。
- Input `240..259` active 状态显示 IP=`192.168.1.223`，DNS/lease 为 0。

### DHCP 成功

测试条件：

- 通过 RTU 临时写 Holding `50=1`。
- 执行 Holding `100=6` 应用网络参数。

测试结论：

- DHCP 成功获取地址 `192.168.1.175`。
- TCP `192.168.1.175:502` 可连接。
- Input `240` flags 中 DHCP active bit 置位。
- Input `254..257` 读到 DNS=`192.168.1.1`。
- Input `258..259` 读到 lease=`43200` 秒。
- Holding `51..63` 仍保持静态 / fallback 参数，不被 DHCP 结果改写。

### DHCP 失败回退

测试条件：

- 通过 RTU 临时写 Holding `50=1`。
- 拔掉网线。
- 执行 Holding `100=6` 应用网络参数。

测试结论：

- 后台命令完成，Holding `103=0`。
- Input `240=1`，表示 TCP 线程运行、DHCP active=0、link down。
- active IP 回退到 `192.168.1.223`。
- DNS/lease 为 0。
- 测试结束后已恢复 Holding `50=0` 并执行 Holding `100=1` 保存。

### 网线插拔

测试结论：

- 网线拔掉后，RTU 仍可读 Input `240..259`。
- 网线拔掉后，Input `240=1`，link/speed/duplex 位清零。
- 网线拔掉后，TCP `192.168.1.223:502` 连接超时，符合预期。
- 网线插回后，Input `240=57`，表示 running/link up/100M/full duplex。
- 网线插回后，TCP `192.168.1.223:502` 恢复可连接。

### TCP 断连重连

测试结论：

- 同一 TCP 长连接连续 80 次读取 Input `240..259` 通过，耗时约 239 ms。
- 20 次短连接读取全部通过，耗时约 10 秒。短连接可用，但不建议作为高频访问方式。
- 空连接后立即关闭、半包后关闭，设备都能恢复监听。
- 客户端连上后长期不发请求时，会临时占用唯一 socket；固件当前通过 5 秒空闲超时释放连接。
- 空闲连接被释放后，新 TCP client 可重新连接并正常读取。

### TCP 端口变更

测试条件：

- 通过 RTU 写 Holding `63=1502`。
- 执行 Holding `100=6` 应用网络参数。

测试结论：

- 应用命令完成，Holding `103=0`。
- TCP `192.168.1.223:502` 超时，旧端口不再监听。
- TCP `192.168.1.223:1502` 可读取 Input `240..259`。
- 通过 RTU 写 Holding `63=502` 并再次执行 Holding `100=6` 后，TCP 恢复到 `192.168.1.223:502`，`1502` 超时。
- 测试结束后已执行 Holding `100=1` 保存参数，Holding `50..63` 最终读回端口 `502`。

### 静态 IP 变更

测试条件：

- DHCP 关闭。
- 通过 RTU 写 Holding `51..54=192,168,1,224`。
- 执行 Holding `100=6` 应用网络参数。

测试结论：

- 应用命令完成，Holding `103=0`。
- TCP `192.168.1.223:502` 超时，旧 IP 不再响应。
- TCP `192.168.1.224:502` 可读取 Input `240..259`。
- Input `242..245` 显示 active IP 为 `192.168.1.224`。
- 通过 RTU 写 Holding `51..54=192,168,1,223` 并再次执行 Holding `100=6` 后，TCP 恢复到 `192.168.1.223:502`，`192.168.1.224:502` 超时。
- 测试结束后已执行 Holding `100=1` 保存参数，Holding `50..63` 最终读回 DHCP=`0`、IP=`192.168.1.223`、端口=`502`。

### 网络参数非法值

测试条件：

- 当前基线 Holding `50..63` 为 DHCP=`0`、IP=`192.168.1.223`、netmask=`255.255.255.0`、gateway=`192.168.1.1`、port=`502`。
- 通过 RTU 分别尝试非法写入。

测试结论：

| 测试项 | 写入动作 | 结果 | 参数是否保持不变 |
| --- | --- | --- | --- |
| DHCP 非法值 | Holding `50=2` | 拒绝 | 是 |
| IP 字节越界 | Holding `51..54=300,168,1,223` | 拒绝 | 是 |
| 半写 IP | Holding `51=192` | 拒绝 | 是 |
| 从 IP 中间写 | Holding `52..54=168,1,224` | 拒绝 | 是 |
| 端口 0 | Holding `63=0` | 拒绝 | 是 |

补充测试：

- Holding `63=1` 可写入。
- Holding `63=65535` 可写入。
- 测试结束后已恢复 Holding `63=502`，并执行 Holding `100=1` 保存。
- 最终 TCP `192.168.1.223:502` 可正常读取 Input `240..259`。

当前异常码说明：

- 上述非法写入均已被拒绝，且参数没有被污染。
- 当前工具读到的异常码多为 Modbus exception code `4`，还没有严格区分为 illegal value。此前已决定暂不优先做严格异常码映射，后续如上位机需要精确错误提示，再把非法地址、非法值、后台忙、资源不足分别映射为更明确的异常码。

### factory MAC 写入和应用

测试条件：

- 原 factory MAC 为 `02:57:50:00:00:02`。
- 通过 RTU 写 Holding `64..69=2,87,80,0,0,126`，即测试 MAC `02:57:50:00:00:7E`。
- 执行 Holding `100=6` 应用网络参数。

测试结论：

- 写入测试 MAC 后，后台命令类型为 `7`，Holding `103=0`。
- Holding `64..69` 可读回 `2,87,80,0,0,126`。
- 执行 Holding `100=6` 后，TCP `192.168.1.223:502` 可恢复连接。
- PC ARP 表显示 `192.168.1.223 -> 02-57-50-00-00-7e`，说明 W5500 active MAC 已切换为测试 MAC。
- 测试结束后写回 Holding `64..69=2,87,80,0,0,2`，再次执行 Holding `100=6`。
- PC ARP 表最终显示 `192.168.1.223 -> 02-57-50-00-00-02`，TCP `192.168.1.223:502` 正常。

注意事项：

- 写 Holding `64..69` 只更新片内 Flash factory 区，不会立刻改变正在运行的 W5500 MAC。
- 必须执行 Holding `100=6` 后，W5500 才会重新应用 factory MAC。
- MAC 改变后，PC 侧 ARP 缓存可能短时间仍指向旧 MAC，表现为 TCP 暂时超时。等待 ARP 重新学习或通过管理员权限清 ARP 后即可恢复。
- factory MAC 必须是有效单播地址；当前测试使用 `02` 前缀的本地管理单播地址。

### TCP 与 RTU 并发压力

测试条件：

- TCP 侧保持一个长连接，连续读取 Input `240..259`。
- RTU 侧通过 COM16 周期读取 Input `240..259`。
- 测试持续 120 秒。
- 测试期间不写参数、不执行后台命令。

测试结论：

- TCP 读取 9201 次，失败 0 次，连接重建 1 次，最终状态 `[57, 23, 192, 168, 1, 223]`。
- RTU 读取 460 次，失败 0 次，最终状态 `[57, 23, 192, 168, 1, 223]`。
- active IP 始终为 `192.168.1.223`。
- Input `240` 始终包含 running/link up/100M/full duplex。
- 测试结束后 Holding `50..63` 读回 DHCP=`0`、IP=`192.168.1.223`、port=`502`。
- PC ARP 表读回 `192.168.1.223 -> 02-57-50-00-00-02`。

## 当前收口判断

网络通信基础健壮性可以认为阶段性收口：

- 静态 IP 正常。
- EEPROM 参数掉电恢复正常。
- DHCP 成功路径正常。
- DHCP 失败回退正常。
- 网线插拔恢复正常。
- TCP 异常断开可恢复。
- 单 socket 空闲占用已有超时保护。
- TCP 端口运行时变更和恢复正常。
- 静态 IP 运行时变更和恢复正常。
- 网络参数非法写入会被拒绝，不会写出半坏参数。
- factory MAC 写入、读回、应用和恢复正常。
- TCP 长连接轮询与 RTU 周期读取并发 120 秒无失败。
