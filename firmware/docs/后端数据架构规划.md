# 数据后端架构计划

本文记录当前后端优先开发方向。项目按工业仪表固件处理，UI、Modbus RTU、Modbus TCP、MSH 调试命令都属于前端；采样、参数、记录、报警和时间状态由后端服务统一维护。

## 分层边界

```text
硬件采集层
  ADC / NTC / HDC1080 / DI / RTC / RS485 / Ethernet / Flash-EEPROM
  ↓
后台服务层
  data_service / backend_request / param_service / record_service / alarm_service
  ↓
前端适配层
  UI / Modbus RTU / Modbus TCP / MSH / 后续 Web 配置
```

前端读取数据时只读 `data_service_get_snapshot()` 或专用查询接口；前端修改状态时提交 `backend_request`，不得直接写 Flash、记录区、报警状态或共享业务结构体。

非浪涌报警按 `非浪涌报警架构设计.md` 处理：活跃报警由 `alarm_service` 维护，计划用 EEPROM KVDB 保存未闭合报警恢复镜像；历史报警改为事件日志，`alarm_start` 和 `alarm_clear` 都由 `record_service` 追加写入片内 Flash TSDB，不在查询路径合并闭合记录。

## 缓存机制

短期继续使用 `device_data_snapshot_t` 作为统一快照。快照由 `data_service` 写入，UI/Modbus/MSH 只读。

当前实现用短临界区复制快照，字段继续增加后建议切换到双缓冲：

```text
data_service 写 back buffer
更新 version/sequence
切换 active buffer
前端读取 active buffer
```

快照建议拆分语义，但对外可以继续合并输出：

```text
realtime      浪涌峰值、NTC、温湿度、DI
status        time_valid、sensor_valid、alarm、rs485、tcp
param_status  参数版本、dirty、持久化状态
diag          update_count、通信计数、错误计数
```

## 请求机制

`backend_request` 是前端写操作的统一入口。当前最小版本支持：

- 设置运行期参数
- 保存参数到 AT24C128 EEPROM KVDB
- 从 AT24C128 重新加载参数
- 恢复运行期默认参数
- 清除运行期记录

后续扩展：

- 设置 RTC 时间
- 清除报警
- 查询/删除历史记录
- 恢复出厂设置
- 网络参数变更后重启协议栈

原则：Modbus 回调和 UI 事件只提交请求并等待结果，不直接做持久化慢操作。

## Modbus 响应及时性规范

Modbus RTU 是强时序前端，寄存器回调必须尽快返回。后续所有 Modbus 相关开发遵守以下规则：

- Modbus 读寄存器只读取 RAM 快照、RAM 参数表或 RAM 状态计数，不允许触发 I2C、ADC 阻塞采样、EEPROM、Flash、网络重启等慢操作。
- Modbus 写普通运行参数只更新 RAM shadow，必须是有界快速操作。
- Modbus 写保存参数、清除记录、恢复默认、设置时间、网络重启等慢操作时，只投递异步请求并快速返回；实际完成状态通过状态寄存器查询。
- EEPROM/Flash 写入不得发生在 Modbus 回调路径内。
- 传感器采样不得发生在 Modbus 回调路径内；Modbus 只能读取 `data_service` 已缓存的快照。
- 如果某个操作最坏情况无法在数毫秒内完成，不得作为同步 Modbus 写回调执行。

当前代码约定：

- `backend_request_set_param_async()` / `backend_request_set_params_async()`：用于 Modbus 等强时序前端，只做参数校验和非阻塞入队；实际 RAM 参数更新由 `backend_req` 线程执行。
- `backend_request_*()` 同步接口：用于 MSH、UI 或显式慢操作请求；EEPROM/Flash 等慢操作仍不得发生在 Modbus 回调路径内。

## Modbus 映射

Modbus RTU 和后续 Modbus TCP 必须共用同一份寄存器映射。当前使用：

```text
app_modbus_rtu.c   协议线程、eMBInit/eMBPoll
app_modbus_map.c   holding/input/coils/discrete 映射
```

正式寄存器表冻结前，当前 holding/input 映射只作为 bring-up 和后端验证使用。

## 已确认硬件映射

- TI1..TI5 业务输入：底层对应 `DI0..DI4` / bit0..bit4，MCU 引脚为 `PA7 / PA6 / PA5 / PA4 / PA3`。业务默认 open 触发报警；当前已确认 open 为 MCU 低电平，close 为 MCU 高电平。
- PC0 / ADC1_IN10：NTC 输入，用户于 2026-06-22 确认。
- PC1 / ADC2_IN11：浪涌输入，用户于 2026-06-22 确认。
