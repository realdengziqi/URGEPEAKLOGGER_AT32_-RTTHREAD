# 存储布局设计

## 目标

本项目采用两类持久化存储：

- AT24C128C EEPROM：用于参数类数据，接入 FlashDB KVDB。
- AT32F403A 片内 Flash：用于报警日志、浪涌记录和 factory 出厂信息；日志接入 FlashDB TSDB，factory 区使用固定镜像。

UI、Modbus、MSH 等前端模块不得直接写 EEPROM、片内 Flash、KVDB 或 TSDB，必须通过 `param_service`、`record_service`、`storage_flashdb` 等后台服务。

## EEPROM 参数区

硬件 EEPROM 为 `AT24C128C-SSHM-T`，容量为 `128 Kbit = 16 KB`，I2C 地址为 `0x50`，页大小为 `64 bytes`。

FAL 设备：

| 项目 | 数值 |
| --- | --- |
| 设备名 | `at24c128` |
| 总容量 | `16 KB` |
| FAL 块大小 | `1 KB` |
| 分区名 | `param_kvdb` |
| 分区范围 | `0x0000..0x3FFF` |
| 数据库 | FlashDB KVDB |

当前 `param_service` 将完整参数镜像保存为 KV：`param_image`。运行期仍由参数服务负责范围校验、默认值、dirty 状态和版本计数。

后续非浪涌报警将增加一个小型 KV：`active_alarm_image`，用于保存 NTC/TI 当前活跃报警恢复镜像。该镜像只在报警发生/结束边沿写入，不周期刷新持续时间，不作为历史日志查询入口。当前最大活跃报警数量按 6 个规划：TI1..TI5 各 1 个，NTC 1 个。

## 片内 Flash 持久化区

AT32F403ARGT7 片内 Flash 总容量按 `1024 KB` 规划。GCC 链接脚本将应用程序区限制为前 `768 KB`，最后 `256 KB` 留给日志数据库和 factory 出厂信息，防止固件镜像覆盖持久化区。

| 区域 | 起始地址 | 大小 | 用途 |
| --- | --- | --- | --- |
| 应用程序 | `0x08000000` | `768 KB` | 固件代码、只读数据、初始化数据 |
| 报警 TSDB | `0x080C0000` | `64 KB` | 报警和普通事件记录 |
| 浪涌 TSDB | `0x080D0000` | `190 KB` | 浪涌事件记录 |
| factory | `0x080FF800` | `2 KB` | 出厂身份信息，当前用于 MAC 地址 |

FAL 设备：

| 项目 | 数值 |
| --- | --- |
| 设备名 | `onchip` |
| 总容量 | `1024 KB` |
| 擦除块大小 | `2 KB` |
| 写入粒度 | `8 bit` |

## FlashDB 分区表

当前分区表定义在 `firmware/external_port/flashdb_port/fal_cfg.h`：

| 分区名 | FAL 设备 | 偏移 | 大小 | 数据库 |
| --- | --- | --- | --- | --- |
| `param_kvdb` | `at24c128` | `0x00000000` | `16 KB` | KVDB |
| `alarm_tsdb` | `onchip` | `0x000C0000` | `64 KB` | TSDB |
| `surge_tsdb` | `onchip` | `0x000D0000` | `190 KB` | TSDB |
| `factory` | `onchip` | `0x000FF800` | `2 KB` | 固定镜像 |

## factory 区

factory 区用于出厂身份信息，不属于用户运行参数，不参与恢复默认参数，也不在 `record clear` 时清除。当前第一版只保存 MAC 地址：

| 字段 | 说明 |
| --- | --- |
| magic | factory 镜像标识 |
| version | 当前为 `1` |
| mac | 6 字节设备 MAC |
| mac_valid | MAC 是否有效 |
| checksum | 镜像校验 |

MAC 地址优先级：

1. factory 区存在有效 MAC 时，W5500 使用 factory MAC。
2. factory 区无效或未写入时，开发阶段回退到 `02:57:50:xx:xx:xx` 本地管理 MAC，后 3 字节由当前 IP 派生。

factory MAC 校验规则：

- 不允许 `00:00:00:00:00:00`。
- 不允许 `FF:FF:FF:FF:FF:FF`。
- 不允许 multicast MAC，即首字节 bit0 必须为 `0`。

量产唯一性由生产工具和台账保证：每台设备分配一个未使用 MAC，写入 factory 区，读回校验，并记录 SN 与 MAC 的绑定关系。

## 当前接入状态

- `param_service_save()`：写入 EEPROM KVDB。
- `param_service_load()`：从 EEPROM KVDB 加载参数镜像。
- `record_service_append_snapshot()`：先写 RAM 环形缓存，再把 TSDB 写入请求投递到 `record_wrq`。
- `record_service_append_manual_test()`：构造完整测试记录后写 RAM，并把 TSDB 写入请求投递到 `record_wrq`。
- `record_wr`：记录持久化线程，优先级 20，栈 1024，通过 FlashDB TSDB 写入 `alarm_tsdb` 或 `surge_tsdb`。
- `alarm_service`：当前维护 NTC/TI 活跃报警 RAM 表；计划通过 EEPROM KVDB 保存 `active_alarm_image`，用于复位恢复未闭合报警。
- `record_service`：报警历史按事件日志写入 TSDB；非浪涌报警发生写 `alarm_start`，报警恢复写 `alarm_clear`，浪涌高阈值报警写 `alarm_instant`。
- `record_service_clear()`：清 RAM 环形缓存，并清理两个 TSDB。
- `factory_service`：读取/写入片内 Flash factory 区，当前用于 W5500 MAC 地址。
- `record status` MSH 命令：查看 RAM 记录数量、TSDB 持久化状态、待写队列、写入成功/失败/丢弃计数。
- `storage` MSH 命令：只读查看 FlashDB 初始化状态和 TSDB 记录数量。
- `factory` MSH 命令：开发和生产阶段写入/读回/清除 factory MAC。

当前实测状态：

```text
record: count=8 capacity=16 total_written=8 next_seq=9 persistent_ready=1
record: persist pending=0 written=8 failed=0 dropped=1
```

其中 `dropped=1` 来自 `record clear` 后的 generation 保护，表示清除记录时丢弃旧队列项，不代表 Flash 写失败。

## 待完善

- 记录服务目前仍以 RAM 环形缓存作为快速查询入口，TSDB 历史查询接口尚未接入 `record_service_get_by_index()`。
- 活跃报警 EEPROM KVDB 镜像尚未实现，当前活跃报警仍只在 RAM 中维护。
- 片内 Flash 日志区首次接入会格式化对应分区，必须在硬件上做掉电、复位、重复写入测试。
- 后续需要增加 TSDB 历史记录分页查询接口，并明确 Modbus TCP/RTU 查询历史记录的状态机。
