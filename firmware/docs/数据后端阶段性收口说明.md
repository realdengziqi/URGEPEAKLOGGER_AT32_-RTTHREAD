# 数据后端阶段性收口说明

本文记录当前“先收口数据后端，再进入 UI 前端联调”阶段已经完成的范围、架构边界、可供 UI 使用的接口能力，以及仍需后续处理的事项。

本文不是最终产品说明书。当前固件仍处于首次开发阶段，Modbus 寄存器表、UI 页面组织和浪涌详情格式后续仍可调整。

## 阶段性结论

当前数据后端可以认为已经达到进入 UI 前端联调的阶段性条件：

- GCC / CMake / J-Link CLI 工具链已经作为主流程使用，Keil 不再是当前开发主路径。
- Modbus RTU 已可通过 RS485 工作，当前测试口为 COM16，默认参数为 `9600 8N1`，从站地址当前为 `2`。
- Modbus RTU 当前串口实现为 USART3 中断收发，不是 DMA，也不是纯轮询。
- Modbus TCP 已基于 W5500 接入，当前使用 socket0，第一版只支持 1 路 TCP client。
- RTU 和 TCP 已共用同一套 Modbus 寄存器映射。
- 参数系统已接入 EEPROM KVDB，运行参数、通信参数、网络参数具备断电保持能力。
- 报警记录已接入片内 Flash TSDB，非浪涌报警按事件日志模型保存，不再做开始/结束合并记录。
- 报警历史分页查询已接入 Modbus，查询路径不会在 Modbus 回调里直接扫 Flash。
- 当前活跃报警已形成后台状态，供前端读取；TI/NTC 报警使能、阈值和消抖参数已经纳入参数体系。
- 网络状态已经通过 Input Register `240..259` 暴露，UI 可以直接显示 TCP、DHCP、链路、IP、网关、DNS、租约等运行状态。
- Modbus TCP 已完成静态 IP、DHCP、DHCP 失败回退、网线插拔、断连重连、端口变更、IP 变更、factory MAC 写入和 RTU/TCP 并发压力的阶段性测试。

因此，下一阶段可以开始 UI 前端页面开发，并用 UI 逐步替代 MSH 和人工 Modbus 测试中的部分观察入口。

## 当前后端边界

项目当前按“硬件采集层 / 后台数据服务层 / 前端交互层”划分。

```text
硬件采集层
  ADC / TI / NTC / RTC / EEPROM / Flash / RS485 / W5500
        ↓
后台数据服务层
  data_service / param_service / record_service / alarm_service / backend_request
        ↓
前端交互层
  UI / Modbus RTU / Modbus TCP / MSH / 后续上位机
```

前端模块的基本规则：

- UI、Modbus RTU、Modbus TCP、MSH 都属于前端。
- 前端读取实时状态时，应读取 `data_service` 快照或对应 service 的查询接口。
- 前端修改参数、清记录、应用通信参数、应用网络参数时，应提交给 `backend_request` 或对应后台 service。
- 前端不得直接写 EEPROM、片内 Flash、TSDB、报警状态结构体或底层采样结果。
- Modbus 回调路径不得执行 EEPROM、Flash、I2C、ADC 阻塞采样、W5500 重启等慢操作。
- 慢操作统一由后台线程异步执行，前端通过状态寄存器或查询接口获取完成结果。

这个边界对 UI 同样适用。UI 不应该因为在本机固件内部运行，就绕过数据服务直接访问底层驱动。

## 已收口模块

### 参数系统

参数由 `param_service` 统一管理，持久化使用 EEPROM 上的 FlashDB KVDB。

当前已经纳入参数体系的主要内容：

- Modbus RTU 地址。
- Modbus RTU 波特率。
- Modbus RTU 数据格式，例如 `8N1`、`8E1`、`8O1`。
- 浪涌报警使能，默认开启。
- TI1..TI5 报警使能 mask，默认关闭。
- TI1..TI5 open 电平配置，默认 open 为低电平。
- TI 报警消抖时间。
- NTC 传感器使能。
- NTC 高温报警使能。
- NTC 高温报警阈值。
- NTC 低温报警使能。
- NTC 低温报警阈值。
- NTC 报警消抖时间。
- 显示亮度。
- 环境温湿度校准偏移。
- DHCP 使能。
- 静态 / fallback IPv4 地址。
- 静态 / fallback 子网掩码。
- 静态 / fallback 网关。
- Modbus TCP 端口。
- factory MAC 写入入口。

通信参数和网络参数采用“写参数不立即生效，必须显式提交”的策略：

- 写 Holding 参数只更新 RAM 参数影子。
- 保存参数需要显式写命令。
- 应用 Modbus RTU 通信参数需要显式写命令。
- 应用网络 / Modbus TCP 参数需要显式写命令。

这样做可以避免上位机写到一半时设备立即切换通信参数，降低现场配置风险。

### 报警系统

非浪涌报警当前主要包括：

- TI1..TI5 开关量报警。
- NTC 高温报警。
- NTC 低温报警。

当前报警模型已经明确为事件日志模型：

- 报警发生写入一条 `alarm_start` 事件。
- 报警恢复写入一条 `alarm_clear` 事件。
- 对外展示时也按独立事件展示，不在查询路径合并成“开始时间 / 结束时间”的单条记录。
- 这样可以保持报警日志按发生时间进入 TSDB，避免为了合并记录而扫描 Flash。

活跃报警用于表达当前仍未恢复的状态：

- TI1..TI5 同一业务输入同一时间最多一个活跃报警。
- NTC 同一时间最多一个活跃报警。
- 当前活跃报警总规模很小，适合使用 EEPROM KVDB 镜像保存。
- 活跃报警恢复时，后台负责更新活跃状态并追加恢复事件。

浪涌报警记录已经有初步格式设计，但浪涌高速采样、积分算法、浪涌详情记录和触发规则仍未最终落地，因此本文不把浪涌详情纳入已收口范围。

### 记录系统

当前片内 Flash 最后 256 KB 用作 FlashDB TSDB 分区，用于记录类数据。

当前已经明确：

- 参数性数据放 EEPROM KVDB。
- 日志性数据放片内 Flash TSDB。
- 报警记录和浪涌详情记录不放同一个 TSDB。
- 非浪涌报警记录先收口分页查询。
- 浪涌详情后续单独设计查询接口。

报警历史查询当前采用分页模式：

- 不做时间范围查询。
- 不做类型过滤。
- 不做原因过滤。
- 上位机或 UI 写翻页控制参数。
- 后台线程完成查询。
- 查询结果缓存在 Input Register 页结果区。

这样可以避免复杂条件查询导致 Flash 扫描压力过大，也更适合当前设备资源。

### Modbus RTU / TCP

当前 Modbus 是主要外部数据前端。

已确认的原则：

- RTU 和 TCP 共用同一套寄存器映射。
- Holding Register 主要用于参数、命令和查询控制。
- Input Register 主要用于实时快照、报警状态、查询结果和网络运行状态。
- RTU/TCP 寄存器映射集中在 Modbus map 层，不应散落到各业务模块。
- Modbus 不直接访问 ADC、Flash、EEPROM、LCD 或 W5500 SPI。

当前 TCP 设计：

- W5500 socket0。
- 1 路 Modbus TCP client。
- 默认静态 IP 为 `192.168.1.223`。
- 默认端口为 `502`。
- DHCP 支持已接入，默认关闭。
- DHCP 成功时，动态地址只作为运行地址，不覆盖静态 / fallback 参数。
- DHCP 失败时回退到静态 / fallback 地址。
- 空闲 TCP 连接 5 秒无完整请求会被主动释放。

网络运行状态已经映射到 Input Register `240..259`，UI 可以直接显示这些状态。

## UI 联调可用数据

UI 下一阶段可以优先绑定以下数据，而不需要等待浪涌高速 ADC 完成。

首页 / 状态总览：

- 当前时间和时间有效标志。
- TI1..TI5 当前状态。
- NTC 当前温度。
- 环境温湿度。
- 当前报警总状态。
- 通信状态摘要。
- 网络状态摘要。

当前报警页：

- TI1..TI5 当前活跃报警。
- NTC 当前活跃报警。
- 报警发生时间。
- 持续时间。
- 报警原因。

报警历史页：

- 使用报警记录分页查询接口。
- 每页最多 4 条记录。
- 先只显示事件列表，不做时间范围筛选。
- `alarm_start` 和 `alarm_clear` 作为独立事件显示。

参数页：

- Modbus RTU 地址、波特率、数据格式。
- TI 报警使能。
- TI open 电平配置。
- TI 消抖时间。
- NTC 传感器使能。
- NTC 高温报警使能和阈值。
- NTC 低温报警使能和阈值。
- NTC 消抖时间。
- DHCP 使能。
- 静态 IP、子网掩码、网关。
- Modbus TCP 端口。

网络状态页：

- TCP 是否运行。
- DHCP 是否 active。
- 网络参数是否正在应用。
- 网线是否连接。
- 速率是否 100M。
- 是否全双工。
- W5500 socket0 状态。
- 当前运行 IP、子网掩码、网关。
- DHCP DNS。
- DHCP lease。

## UI 联调建议顺序

建议 UI 前端按以下顺序接入，避免一开始就把页面、参数写入、报警记录和浪涌详情全部混在一起：

1. 首页只读快照：先显示时间、TI、NTC、环境温湿度、报警总状态、网络摘要。
2. 网络状态只读页：直接绑定 Input `240..259` 对应的后台状态。
3. 当前报警只读页：显示当前活跃报警，不做确认、清除或复杂操作。
4. 报警历史只读页：接入分页查询，只做上一页、下一页、第一页、指定页。
5. 参数查看页：先只读显示当前参数。
6. 参数修改页：再接入编辑、提交、保存、应用命令。
7. 蜂鸣器短鸣反馈：按键短按或参数提交成功时调用后端蜂鸣器接口。

浪涌详情页建议后置，等高速 ADC、积分算法、浪涌记录格式和详情查询接口明确后再做。

以下内容仍不能宣布最终收口：

- 浪涌高速 ADC 采样。
- 罗氏线圈积分算法。
- 浪涌触发阈值、记录阈值和报警阈值的最终业务规则。
- 浪涌详情记录格式。
- 浪涌详情分页 / 定位查询接口。
- 最终产品版 Modbus 寄存器表冻结。
- 严格 Modbus 异常码映射。
- 生产阶段 factory MAC 分配脚本和烧录校验流程。
- 更长时间老化测试。
- UI 前端和数据后端的完整联调。
- LCD 刷新对通信、采样和蜂鸣器的影响验证。
