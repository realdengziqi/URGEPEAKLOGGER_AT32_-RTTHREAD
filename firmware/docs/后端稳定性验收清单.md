# 后端稳定性验收清单

本文记录当前“数据后端优先”阶段的稳定性门槛。目标是在 UI 继续开发前，先确认数据采集、参数、记录、Modbus 和 RTOS 调度边界足够稳定。

## 当前结论

- GCC/CMake/J-Link CLI 工具链已经可作为主工作流继续使用。
- AT24C128C EEPROM 已接入 FlashDB KVDB，用于参数镜像 `param_image`。
- AT32F403A 片内 Flash 最后 256 KB 已划分为 FlashDB TSDB，用于报警和浪涌记录。
- 记录写入已经改为 `record_wr` 线程异步写 TSDB，前端追加记录时不直接执行 Flash 写入。
- Modbus RTU 当前使用 USART3 中断方式，不是 DMA，也不是纯轮询。
- UI、Modbus、MSH 均按前端处理，原则上只通过 `data_service`、`param_service`、`record_service`、`backend_request` 访问后端。
- Modbus TCP / W5500 网络通信已完成静态 IP、DHCP 成功、DHCP 失败回退、网线插拔、TCP 断连重连阶段性收口，详细记录见 `网络通信收口测试记录.md`。
- 当前数据后端阶段性收口结论和 UI 联调入口见 `数据后端阶段性收口说明.md`。

## RT-Thread 线程实测

2026-06-23 通过 COM12 执行 `sys info`、`sys threads` 的结果：

```text
mcu=AT32F403ARGT7 rtos=RT-Thread tick_hz=1000 prio_max=32
finsh name=tshell prio=20 stack=2048 cmd=96 arg_max=8

thread           prio stat     stack used/size
tshell             20 run        612/2048
mb_rtu             16 suspend    360/1024
backend_req        17 suspend    412/1024
record_wr          20 suspend    376/1024
data_svc           18 suspend    468/768
tidle0             31 ready      104/512
timer               4 suspend    156/512
main               10 suspend   1100/2048
```

说明：

- `suspend` 多数表示线程正在等待消息队列、协议事件或周期延时，是正常状态。
- `tshell` 在执行命令时为 `run`，正常。
- `data_svc` 当前栈剩余约 300 字节，后续增加采样融合、报警判断或更多日志时建议把栈从 768 增加到 1024。

## Modbus RTU 串口实现

当前 Modbus RTU 使用 `firmware/external_port/freemodbus_port/portserial.c`：

- UART：USART3。
- 物理接口：RS485，当前测试口为 COM16。
- 默认参数：9600 8N1，slave 地址当前为 2。
- RX：`USART_RDBF_INT` 中断接收，写入 128 字节环形缓冲。
- TX：`USART_TDBE_INT` 中断逐字节发送。
- RS485 DE：PA15 控制，发送完成等待 `USART_TDC_FLAG` 后切回接收。
- 当前未使用 DMA。

## 已完成测试

| 项目 | 方法 | 结果 |
| --- | --- | --- |
| 参数保存 | `param set/save/load/get/status` | EEPROM KVDB 保存和恢复通过 |
| 记录保存 | `record add/status`、复位后查询 | TSDB 写入和恢复通过 |
| Modbus Input 读并发记录写 | COM16 读 16 个 input register，同时注入 12 条记录 | 120 次读，0 timeout |
| Modbus Holding 读写并发记录写 | COM16 读 7 个 holding register，写亮度参数，同时注入 8 条记录 | 80 次读、40 次写，0 timeout |
| RTOS 线程状态 | COM12 `sys threads` | 当前栈余量可接受 |

## 后端服务当前状态样例

```text
backend: data_update=2176 time_valid=1 sensor_flags=0x0f
backend: param_count=7 param_ver=86 dirty=1 persistent=1
backend: record_count=8/16 total_written=8 persistent=1

record: count=8 capacity=16 total_written=8 next_seq=9 persistent_ready=1
record: persist pending=0 written=8 failed=0 dropped=1

modbus: running=1 poll=988 holding_rd=164 holding_wr=84 input_rd=80
modbus: holding[1..17]=runtime params input[1..101]=data/alarm/record snapshot
```

备注：

- `param dirty=1` 表示测试期间改过运行参数但尚未再次 `param save`，不是存储故障。
- `record dropped=1` 来自 `record clear` 时丢弃旧队列项的保护逻辑，不是写失败。

## 稳定迁移判定门槛

宣布可以稳定迁移到 GCC/CMake/J-Link CLI 前，建议至少满足：

1. `tools/build_cmake.ps1` 连续构建通过。
2. `tools/flash_jlink.ps1` 可稳定烧录，复位后 COM12 日志正常。
3. `sys threads` 中所有业务线程存在，且栈余量合理。
4. `param save/load` 在断电或复位后仍能恢复参数。
5. `record add` 在复位后仍能从 TSDB 恢复记录。
6. Modbus RTU 在参数读写和记录写入并发时无 timeout。
7. Modbus 回调路径不执行 EEPROM、Flash、阻塞 ADC、I2C 慢操作。
8. 文档中的寄存器、存储分区、线程职责与代码保持一致。

## 后续优先事项

3. 增加 Modbus 通信错误计数和最近错误状态，方便现场判断 RS485 链路质量。

5. 绕过浪涌高速 ADC 之前，先把低速传感器、参数、记录、Modbus 的后端链路验收稳定。
