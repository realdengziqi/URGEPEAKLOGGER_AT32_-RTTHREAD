# uGUI 集成与控件库复用方案

## 目标

本项目计划引入 uGUI 作为 240x240 彩屏 UI 的基础绘图库，并在其上建立一套项目自有控件样式。后续该控件库应能作为 Git submodule 被多个项目复用，以保持工业仪表类产品的显示风格一致。

本方案只定义集成边界和迁移步骤，不直接修改 LCD 引脚、ST7789 驱动、RT-Thread 配置或 Keil Target。

## 第三方来源

- 原始库名称：uGUI / µGUI
- 上游仓库：https://github.com/achimdoebler/UGUI
- 用途：嵌入式平台无关 GUI 绘图、窗口和基础控件框架
- 当前上游状态：GitHub 仓库未发布 release，主分支为 `master`
- 核心文件：`ugui.c`、`ugui.h`、`ugui_config.h`
- 许可证：上游 `LICENSE.md` 允许个人、非营利和商业产品使用、修改和再发布，但要求源码再发布时保留版权声明；无质量保证
- 待确认：正式集成时应固定具体 commit，避免直接跟随 `master`

## 推荐目录结构

```text
project/
  external/
    ugui/                         # uGUI 上游原始副本，建议作为 submodule
    surge-ui-kit/                 # 自研控件库，后续建议独立仓库并作为 submodule
  firmware/
    vendor/
      ugui/                       # 从 external/ugui 复制的编译副本
        SOURCE_NOTICE.md
        ugui.c
        ugui.h
        ugui_config.h
    external_port/
      ugui_port/
        ugui_port.c               # uGUI 到 screen_st7789 的适配层
        ugui_port.h
    app/
      inc/
        app_ui.h
      src/
        app_ui.c                  # 本项目页面逻辑，只调用控件库和 data_service
```

如果先不拆独立仓库，可先在本项目中放置：

```text
firmware/app/inc/surge_ui_style.h
firmware/app/inc/surge_ui_widgets.h
firmware/app/src/surge_ui_widgets.c
```

待控件 API 稳定后，再迁移到独立的 `surge-ui-kit` 仓库。

## 分层边界

```text
app_ui 页面层
  ↓
surge-ui-kit 自研控件和样式层
  ↓
uGUI 原始绘图库
  ↓
ugui_port 显示适配层
  ↓
screen_st7789 驱动
```

各层职责：

- `screen_st7789`：只负责 LCD 初始化、背光、像素/矩形/字符等底层绘制能力，不包含产品 UI 逻辑。
- `ugui_port`：只实现 uGUI 需要的像素输出、屏幕尺寸、初始化绑定，不读取业务数据。
- `uGUI`：保持第三方源码原样，不直接修改。需要裁剪时优先通过 `ugui_config.h` 或项目适配层完成。
- `surge-ui-kit`：定义颜色、字体、间距、状态块、数值卡片、页眉、页脚、报警条、状态标签等统一控件。
- `app_ui`：只组织页面和读取 `data_service_get_snapshot()`，不直接访问 ADC、Flash、Modbus 或 LCD 硬件细节。

## 控件库建议

优先抽象工业仪表常用控件，而不是一开始做通用 UI 框架：

- `surge_ui_draw_header()`：标题、页码、时间状态。
- `surge_ui_draw_footer()`：按键提示、通信状态提示。
- `surge_ui_draw_value_tile()`：标签 + 大号数值 + 单位。
- `surge_ui_draw_status_row()`：状态名 + 状态值 + 状态颜色。
- `surge_ui_draw_alarm_banner()`：报警/正常状态提示条。
- `surge_ui_draw_progress_bar()`：记录容量、信号质量或占比类显示。

控件 API 应只接收普通值和字符串，不直接读取 data service：

```c
void surge_ui_draw_value_tile(rt_uint16_t x,
                              rt_uint16_t y,
                              rt_uint16_t w,
                              const char *label,
                              const char *value,
                              const char *unit,
                              rt_uint16_t value_color);
```

这样控件库可以复用到其他项目，不绑定本项目的 `device_data_snapshot_t`。

## Git submodule 策略

建议分两类 submodule：

```text
external/ugui
external/surge-ui-kit
```

原因：

- uGUI 是第三方上游原始副本，应尽量不修改。
- `surge-ui-kit` 是我们自己的风格和控件库，应该可独立演进和复用。
- 本项目参与编译的第三方源码仍应复制到 `firmware/vendor` 或 `firmware/external_port`，不直接从 `external` 编译。

正式新增 submodule 前需要确认：

- uGUI 上游 URL、固定 commit、许可证接受情况。
- `surge-ui-kit` 使用本地仓库、Gitea 仓库还是 GitHub 私有仓库。
- 控件库许可证或内部使用声明。

## 集成步骤

1. 确认 uGUI 上游仓库、许可证和固定 commit。
2. 执行 `git submodule add https://github.com/achimdoebler/UGUI.git external/ugui`。
3. 从 `external/ugui` 复制 `ugui.c`、`ugui.h`、`ugui_config.h` 到 `firmware/vendor/ugui`，并添加 `SOURCE_NOTICE.md`。
4. 新增 `firmware/external_port/ugui_port`，把 uGUI 像素输出绑定到 `screen_st7789_draw_pixel()` 或更高效的区域刷写接口。
5. 将 uGUI 文件和 port 文件加入 Keil 工程。
6. 先实现一个最小页面：初始化 uGUI、清屏、绘制页眉和一个数值块。
7. 将现有 `app_ui.c` 中直接调用 `screen_st7789_*` 的页面绘制逐步替换为控件库调用。
8. 硬件验证 LCD 初始化、颜色、坐标方向、字符显示、按键翻页、刷新耗时。
9. 控件 API 稳定后，抽出 `surge-ui-kit` 独立仓库，再作为 submodule 引入其他项目。

## Keil 工程影响

预计新增编译文件：

- `firmware/vendor/ugui/ugui.c`
- `firmware/external_port/ugui_port/ugui_port.c`
- `firmware/app/src/surge_ui_widgets.c` 或后续控件库源码

预计新增 include path：

- `firmware/vendor/ugui`
- `firmware/external_port/ugui_port`
- 控件库头文件目录

不应修改：

- MCU 型号
- startup file
- linker script
- system clock / PLL
- LCD 引脚和 SPI2 配置
- RT-Thread tick、heap、线程优先级

## RT-Thread 与 UI 刷新要求

- uGUI 绘图只能在线程上下文调用，不能在 ISR 中调用。
- UI 线程应通过脏标志或页面状态机局部刷新，避免高频全屏刷新。
- 初期可以继续由现有 `app_ui_poll()` 驱动按键翻页，后续再评估是否独立 `ui_thread`。
- UI 层继续通过 `data_service_get_snapshot()` 获取显示数据。
- 控件库不得直接访问 ADC、Flash、Modbus、RTC 底层驱动。

## 硬件验证点

uGUI 接入后必须在实物上确认：

- ST7789 初始化后屏幕方向和坐标是否正确。
- RGB565 颜色是否正确。
- 文本和控件是否越界或重叠。
- 页面刷新耗时是否影响按键响应和通信。
- 背光控制是否仍可用。
- `lcd init`、`lcd demo`、`lcd ui` 等 bring-up 命令是否需要迁移或保留旧路径。

## 待确认问题

- 是否接受 `achimdoebler/UGUI` 的许可证条款。
- uGUI 固定到哪个 commit。
- `surge-ui-kit` 是否立即创建独立仓库，还是先在本项目内孵化。
- 控件库后续是否需要支持非 240x240 分辨率。
- 是否需要中文字体；如果需要，应单独确认字库来源、授权和 Flash/RAM 占用。
