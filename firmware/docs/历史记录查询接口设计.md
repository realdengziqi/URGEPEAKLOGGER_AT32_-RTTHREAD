# 历史记录查询接口设计

## 设计结论

第一版历史记录查询只做翻页查询，不做时间范围查询，不做类型过滤，不做报警原因过滤。

本固件仍处于开发阶段，尚未对外发行，不保留旧报警闭合记录格式兼容。报警历史正式采用事件日志模型，`event_action` 是明确字段，不使用旧的 `end_timestamp_ms` / `end_datetime` 字段。

推荐查询方式：
- 读取最近一页。
- 按记录序号向前/向后翻页。
- 主站拿到记录后自行显示、筛选或缓存。

这样做的原因：
- `sequence` 是设备内部单调递增的记录编号，比 RTC 时间更可靠。
- 时间可能无效、未校时或被用户修改，不适合作为第一版查询主键。
- 时间范围查询和类型过滤如果落到 FlashDB TSDB，通常需要遍历，容易影响 Modbus 响应及时性。
- 翻页查询更符合工业仪表和 Modbus 主站的常见实现方式。

## 当前边界

本设计只冻结：
- 查询方式：按序号翻页。
- 通用记录头。
- 非浪涌报警记录字段。
- 浪涌详情记录第一版字段规划。
- 浪涌报警记录到浪涌详情记录的引用关系。
- Modbus 临时查询窗口。

本设计不冻结：
- TSDB 全量扫描机制。
- 时间范围查询。
- 类型过滤。
- 报警原因过滤。

## 当前存储现状

当前 `record_service` 已经实现：
- RAM 环形缓存：`RECORD_SERVICE_CAPACITY = 16`。
- TSDB 持久化：报警记录写入 alarm TSDB，浪涌记录写入 surge TSDB。
- 启动恢复：从 alarm TSDB 和 surge TSDB 反向恢复最近若干条到 RAM 缓存。
- 写入线程：`record_wr` 异步写 TSDB。

当前限制：
- RAM 缓存只保留最近 16 条。
- 已实现报警历史 `alarm_tsdb` 的分页查询入口，当前先用于 MSH 调试验证。
- 尚未实现浪涌详情 `surge_tsdb` 的分页查询。
- `record_service_item_t` 当前同时承载报警、浪涌和手动测试字段；浪涌字段不是正式格式。

RAM 查询仍只覆盖最近 16 条。报警深历史分页走 `alarm_tsdb`，并且只返回 `RECORD_SERVICE_EVENT_ALARM`，不会把开发用 `manual_test` 记录混入报警历史。浪涌详情记录和报警记录不是同一个存储集合，浪涌详情后续单独通过 `surge_tsdb` 查询。

## 记录类型

记录类型继续使用 `record_service_event_type_t`：

| 类型 | 含义 | 当前状态 |
| --- | --- | --- |
| `RECORD_SERVICE_EVENT_SURGE` | 浪涌详情记录 | 第一版字段见 `浪涌报警事件记录格式.md` |
| `RECORD_SERVICE_EVENT_ALARM` | 报警事件记录，包含报警发生、报警恢复和浪涌瞬时报警 | 非浪涌 start/clear 事件见 `非浪涌报警记录格式.md`；浪涌报警需要引用对应浪涌详情 |
| `RECORD_SERVICE_EVENT_MANUAL_TEST` | 开发/人工测试记录 | 仅用于调试 |

第一版查询不会按类型过滤，查询结果中只返回 `type`，由主站自行判断。

## 通用记录头

所有记录都应具有以下通用字段：

| 字段 | 含义 |
| --- | --- |
| `sequence` | 记录序号，单调递增 |
| `format_version` | 记录格式版本 |
| `type` | 记录类型 |
| `timestamp_ms` | 发生 tick 时间 |
| `time_valid` | RTC 时间是否有效 |
| `datetime` | 发生 RTC 时间 |

对于报警事件日志，事件时间统一放在 `timestamp_ms` / `datetime` 中。`duration_ms` 只在 `alarm_clear` 事件中表示本次报警持续时间，`alarm_start` 和 `alarm_instant` 中为 0。

| 字段 | 含义 |
| --- | --- |
| `event_action` | 报警事件动作：`alarm_start`、`alarm_clear`、`alarm_instant` |
| `duration_ms` | `alarm_clear` 事件中的持续时间，其他事件为 0 |

## 非浪涌报警记录字段

非浪涌报警历史按事件日志保存，不合并为闭合记录。先固定以下字段：

| 字段 | 含义 |
| --- | --- |
| `alarm_id` | 报警 ID |
| `event_action` | `alarm_start` 或 `alarm_clear` |
| `alarm_reason` | 报警原因 |
| `timestamp_ms` | 报警事件 tick 时间 |
| `datetime` | 报警事件 RTC 时间 |
| `duration_ms` | `alarm_clear` 事件中的持续时间，`alarm_start` 为 0 |
| `ntc_temp_milli_c` | 事件发生时 NTC 温度 |
| `env_temp_milli_c` | 事件发生时环境温度 |
| `env_humi_milli_percent` | 事件发生时环境湿度 |
| `digital_input_state` | 事件发生时 TI/DI 原始输入状态 |
| `alarm_state` | 事件发生时报警状态 |

报警原因当前包括：
- NTC 高温
- NTC 低温
- TI1 open
- TI2 open
- TI3 open
- TI4 open
- TI5 open

第一版查询不会按报警原因过滤。

展示侧也按事件日志展示，例如：

```text
TI1 开路报警发生
TI1 开路报警恢复，持续 258s
```

设备端不在查询路径按 `alarm_id` 合并 start/clear，不为了展示闭合记录而扫描 Flash。

## 浪涌详情记录字段

浪涌详情记录按瞬时事件处理，不维护类似 NTC/TI 的当前活跃报警。第一版完整字段见 `浪涌报警事件记录格式.md`。

浪涌采用两级阈值：

- 低阈值 `record_current_milli_ka`：超过后进入浪涌详情记录。
- 高阈值 `alarm_current_milli_ka`：超过后额外生成浪涌报警记录。

因此，低阈值记录不一定是报警；高阈值报警必须能反查对应的浪涌详情。

通用历史查询窗口应至少能返回：

| 字段 | 含义 |
| --- | --- |
| `record_current_milli_ka` | 记录准入阈值 |
| `alarm_current_milli_ka` | 报警阈值 |
| `alarm_triggered` | 本条浪涌详情是否触发报警 |
| `linked_alarm_id` | 对应浪涌报警记录 ID，未触发报警时为 0 |
| `waveform_type` | 波形/线圈类型，`8/20us` 或 `10/350us` |
| `flags` | 标定有效、极性、超量程、raw-only 等标志 |
| `peak_current_milli_ka` | 标定有效后的浪涌峰值 |
| `peak_raw` | 峰值 ADC raw |
| `baseline_raw` | 触发前基线 raw |
| `peak_delta_raw` | 峰值相对基线差值 |
| `front_time_ns` | 波前时间 |
| `half_value_time_ns` | 半峰值时间 |
| `sample_rate_khz` | 采样率 |
| `pulse_count` | 当前事件内脉冲数量 |

在罗氏线圈参数和高速 ADC 积分算法确认前：

- 可以记录 raw、触发状态、采样率和窗口信息。
- `calibration_valid=0` 时，不输出伪造的真实 kA 峰值。
- 当前 `peak_milli_kv` 等调试字段不能继续作为正式浪涌记录格式。

## 浪涌报警记录字段

浪涌报警记录用于报警页和上位机报警摘要，不重复保存完整浪涌详情。推荐字段为：

| 字段 | 含义 |
| --- | --- |
| `alarm_id` | 报警 ID |
| `timestamp_ms` / `datetime` | 报警发生时间 |
| `alarm_reason` | 固定为浪涌过流报警 |
| `alarm_current_milli_ka` | 本次触发报警使用的阈值 |
| `peak_current_milli_ka` | 可选摘要，未标定时为 0 |
| `surge_sequence` | 对应浪涌详情记录的 sequence |
| `surge_event_id` | 对应浪涌详情记录的 event_id |
| `surge_tsdb_time` | 可选内部索引，未使用为 0 |

报警原因显示建议表达为：

```text
浪涌峰值超过 X.X kA
```

`surge_sequence` 和 `surge_event_id` 是对外稳定引用；裸 Flash 地址或 FlashDB 内部位置只能作为后端内部优化字段，不应作为正式 Modbus 协议字段。

## record_service 查询接口规划

当前已经提供以下 RAM 最近记录查询接口：

```c
rt_err_t record_service_query_latest_page(rt_size_t offset_from_latest,
                                          rt_size_t max_count,
                                          record_service_item_t *items,
                                          rt_size_t *actual_count);

rt_err_t record_service_query_page_by_sequence(rt_uint32_t start_sequence,
                                               rt_size_t max_count,
                                               record_service_item_t *items,
                                               rt_size_t *actual_count);
```

说明：
- `query_latest_page(0, ...)` 表示从最新记录开始取一页。
- `offset_from_latest` 用于向旧记录方向翻页。
- `query_page_by_sequence(start_sequence, ...)` 表示从指定序号开始取一页。
- 第一阶段只查 RAM 最近 16 条。
- 如果请求的序号不在 RAM 缓存中，返回明确失败状态，暂不扫描 TSDB。

当前已经提供以下报警深历史分页接口：

```c
rt_err_t record_service_query_alarm_latest_page(rt_size_t offset_from_latest,
                                                rt_size_t max_count,
                                                record_service_item_t *items,
                                                rt_size_t *actual_count,
                                                rt_size_t *total_count);
```

说明：
- 该接口只访问 `alarm_tsdb`，不访问 `surge_tsdb`。
- 该接口只返回 `RECORD_SERVICE_EVENT_ALARM`，过滤掉开发测试记录。
- 返回顺序为从最新到最旧。
- `offset_from_latest` 表示从最新报警事件开始跳过多少条。
- `total_count` 当前返回本次查询已扫描到的有效报警事件数量；当本页取满时，它通常等于 `offset_from_latest + actual_count`，不代表精确全库总数。
- 这样做是为了避免调试命令为统计总量长时间扫描整块 Flash；正式协议如果需要精确总数，应放到异步查询或后台索引中实现。
- 当前先用于 MSH 调试验证，不接入 Modbus 同步回调。

当前 Modbus 已提供两类报警历史查询入口：

- 单条查询窗口：Holding `120..139`，Input `120..159`。其中 Holding `120=3` 表示按报警历史 offset 查询，`121` 为 offset，写 `124=1` 投递查询。该窗口适合主站逐条兼容读取。
- 报警列表分页窗口：Holding `140..149`，Input `160..239`。该窗口按页返回最多 4 条报警事件，适合报警页列表显示。

两类入口都遵守同一个原则：由后台线程访问 TSDB，不允许 Modbus 回调直接遍历 Flash。

## Modbus 查询窗口规划

临时规划 Holding `120..139` 作为历史记录单条查询控制区，Input `120..159` 作为单条查询结果窗口；Holding `140..149` 作为报警列表分页控制区，Input `160..239` 作为报警列表页结果窗口。正式寄存器表冻结前可调整。

### Holding Register 120..139：查询控制

| 地址 | 读写 | 含义 |
| --- | --- | --- |
| 120 | R/W | 查询模式：`0` 空闲，`1` 最近页，`2` 按序号页 |
| 121 | R/W | 页内索引或最近 offset，模式 1 有效 |
| 122 | R/W | 查询起始序号高 16 位，模式 2 有效 |
| 123 | R/W | 查询起始序号低 16 位，模式 2 有效 |
| 124 | W | 写 `1` 触发查询 |
| 125 | R | busy，第一阶段通常为 0 |
| 126 | R | result，0 成功，负数表示失败 |
| 127 | R | 查询结果有效：0 无效，1 有效 |
| 128 | R | 当前 RAM 缓存记录数 |
| 129 | R | 最新 sequence 高 16 位 |
| 130 | R | 最新 sequence 低 16 位 |
| 131 | R | 最旧 sequence 高 16 位 |
| 132 | R | 最旧 sequence 低 16 位 |
| 133 | R | 本次查询绑定的记录区 generation 高 16 位 |
| 134 | R | 本次查询绑定的记录区 generation 低 16 位 |
| 135 | R | 本次查询返回条数，当前单条窗口为 `0` 或 `1` |
| 136..139 | R | 保留 |

说明：
- 第一阶段可以同步快速完成，因为只查 RAM 最近 16 条。
- 每次清空记录时，记录区 generation 会递增；如果清记录发生在查询之后，Input `120..159` 查询结果窗口自动失效，主站应以 `127` 的有效标志为准。
- 后续扩展 TSDB 深历史时，`busy/result` 语义保持不变。
- 不增加类型过滤、时间范围过滤和报警原因过滤寄存器。

### Input Register 120..159：查询结果

| 偏移 | 含义 |

| 1 | sequence 高 16 位 |
| 2 | sequence 低 16 位 |
| 3 | format_version |
| 4 | alarm_id 高 16 位 |
| 5 | alarm_id 低 16 位 |
| 6 | alarm_reason |
| 7 | timestamp_ms 高 16 位 |
| 8 | timestamp_ms 低 16 位 |
| 9 | event_action：`0` none，`1` alarm_start，`2` alarm_clear，`3` alarm_instant |
| 10 | source_index：NTC 为 0，TI1..TI5 为 1..5，浪涌为 10 |
| 11 | duration_ms 高 16 位 |
| 12 | duration_ms 低 16 位 |
| 13 | event year |
| 14 | event month/day |
| 15 | event hour/minute |
| 16 | event second/weekday |
| 17..20 | 保留，读 0 |
| 21 | ntc_temp，单位 0.01 摄氏度，int16 |

| 24 | digital_input_state << 8 | alarm_state |

### Holding Register 140..149：报警列表分页控制

该窗口只查询 `alarm_tsdb` 中的报警事件日志，不查询浪涌详情 `surge_tsdb`，不做时间范围、报警原因或类型过滤。

| 地址 | 读写 | 含义 |
| --- | --- | --- |
| 140 | R/W | 翻页命令：`0` 空闲，`1` 加载指定页，`2` 下一页，`3` 上一页，`4` 第一页 |
| 141 | R/W | 指定页码，0 表示最新页 |
| 142 | R/W | 页大小，当前最大 4；写 0 使用默认 4 |
| 143 | W | 写 `1` 触发分页查询 |
| 144 | R | busy：1 正在查询，0 空闲 |
| 145 | R | done：1 已完成至少一次查询 |
| 146 | R | result：0 成功，负数按 int16 解释 |
| 147 | R | current_page：当前页码 |
| 148 | R | item_count：当前页返回条数 |
| 149 | R | flags：bit0=has_prev，bit1=has_next |

查询由 `mb_apage` 后台线程完成。`has_next` 通过内部多查 1 条判断，对外最多返回 4 条记录。

### Input Register 160..239：报警列表分页结果

前 12 个寄存器为页状态头，后面 4 个记录槽，每槽 17 个寄存器。

| 地址 | 含义 |

| 161 | done |

| 164 | page_size |
| 165 | item_count |
| 166 | flags：bit0=has_prev，bit1=has_next |
| 167 | generation 高 16 位 |
| 168 | generation 低 16 位 |
| 169 | 最近翻页命令 |
| 170 | 最近请求页码 |
| 171 | 保留，读 0 |

记录槽：

| 槽号 | 地址范围 |
| --- | --- |
| 0 | 172..188 |
| 1 | 189..205 |
| 2 | 206..222 |
| 3 | 223..239 |

槽内 17 个字段沿用 Input `120..136` 的通用报警摘要字段：valid/type、sequence、format_version、alarm_id、alarm_reason、timestamp_ms、event_action、source_index、duration_ms 和 RTC 时间。

## 并发和实时性要求

- Modbus 回调不能遍历 TSDB。
- 第一阶段只读 RAM 缓存，允许在 Modbus 回调中快速完成。
- 后续如果访问 Flash，必须改为异步查询，由 record service 线程处理。
- 清记录时必须让查询结果失效。
- 记录写入和记录查询需要由 `record_service` 保护数据一致性。

## 分阶段实现建议

第一阶段：
- 增加 RAM 最近页查询接口。
- Modbus 支持查询最近 16 条中的任意一条或一页。
- MSH 增加查询命令辅助调试。

第二阶段：
- 增加按 `sequence` 查询 RAM 缓存。
- 查询不到时返回明确状态，暂不扫 TSDB。

第三阶段：
- 报警历史已经具备 `alarm_tsdb` 最新页分页调试入口，后续需要补齐正式异步查询状态机和协议映射。
- 浪涌详情 `surge_tsdb` 分页待浪涌记录正式字段冻结后再实现。
- 查询仍按翻页模型，不做时间范围查询。
- Modbus 访问 TSDB 查询必须异步执行。

第四阶段：
- 浪涌采样和积分方案确认后，实现 `浪涌报警事件记录格式.md` 中的详情字段和报警引用字段。
- 更新 Modbus 查询结果保留区。

## 明确不做的内容

第一版不做：
- 时间范围查询。
- 类型过滤。
- 报警原因过滤。
- Modbus 回调内 TSDB 遍历。

如后续客户强需求过滤，建议由上位机基于翻页结果自行过滤；设备端只提供稳定、顺序、可分页的原始记录流。

## 待确认问题

- 正式历史容量目标是多少。
- 是否需要导出全部记录。
- 浪涌详情记录和浪涌报警引用格式已完成第一版设计，真实 kA 换算仍待罗氏线圈参数和高速采样算法确认。
