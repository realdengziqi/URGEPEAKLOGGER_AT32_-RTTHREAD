# 浪涌报警事件记录格式

本文定义第一版浪涌报警事件的数据格式。设计依据为竞品手册中“触发电流、浪涌信息、多脉冲记录”等功能描述，并结合本项目当前硬件状态：外部罗氏线圈无积分器，后续真实测量路线为高速 ADC 采样加数字积分。

## 设计结论

浪涌报警不按 NTC/TI 那样处理为“当前活跃报警”。浪涌是一个瞬时事件：

- 当积分后的浪涌电流峰值达到记录阈值时，生成一条浪涌详情记录。
- 当同一条浪涌详情记录的峰值达到报警阈值时，再生成一条浪涌报警记录。
- 浪涌详情记录写入 `surge_tsdb`，浪涌报警记录写入报警记录区，并通过逻辑引用指向对应的浪涌详情。
- 蜂鸣器、LED、UI 或 Modbus 可以基于“最近发生过浪涌事件”做提示，但后端不维护浪涌活跃报警槽。
- 浪涌报警默认开启；后续如客户要求，可增加独立使能参数。

## 竞品逻辑判断与本项目取舍

从竞品手册当前可见文字判断，竞品没有明确说明“低于触发电流是否也进入浪涌记录”：

- 手册把开关量输入描述为“通道报警信息”，并明确记录报警通道编号、报警日期和时间。
- 手册把罗氏线圈输入描述为“浪涌信息”或“雷击信息”，记录浪涌次数、极性、峰值、波前时间、半峰值时间、Q 值、能量、电流陡度和发生日期/时间。
- “触发电流”紧跟在浪涌信息字段之后，选项为 `0.5 / 1 / 2.5 / 5 kA`，更符合“浪涌记录准入阈值”的含义。
- 手册没有看到“低于触发电流也进入浪涌记录，但不算报警”的描述。

但从本项目产品体验和后续 UI 区分需求看，建议采用“两级阈值”：

| 阈值 | 用途 |
| --- | --- |
| `record_current_milli_ka` | 低阈值，超过后进入浪涌详情记录 |
| `alarm_current_milli_ka` | 高阈值，超过后额外生成浪涌报警记录 |

推荐判定逻辑：

```text
如果 abs(peak_current_milli_ka) >= record_current_milli_ka：
    生成浪涌详情记录
    浪涌记录次数 +1
    如果 abs(peak_current_milli_ka) >= alarm_current_milli_ka 且浪涌报警使能：
        在浪涌详情记录中标记 alarm_triggered
        生成浪涌报警记录
        浪涌报警记录引用该浪涌详情记录
    否则：
        在浪涌详情记录中标记未触发报警
否则：
    不生成正式浪涌详情记录
```

这样可以支持“记录阈值 0.5 kA，报警阈值 5 kA”的场景：检测到 2 kA 时进入浪涌详情记录，但不进入报警页；检测到 6 kA 时同时进入浪涌详情记录和报警页。后续 UI 可以根据浪涌详情记录中的 `alarm_triggered` 标志使用不同颜色显示。

第一版触发规则建议对齐竞品：

| 参数 | 建议 |
| --- | --- |
| 记录电流选项 | `0.5 / 1 / 2.5 / 5 kA` |
| 默认记录电流 | `0.5 kA`，待确认 |
| 报警电流选项 | `0.5 / 1 / 2.5 / 5 kA` |
| 默认报警电流 | `5 kA` |
| 显示和协议分辨率 | `0.1 kA` |
| T1 波形 | `10/350 us` |
| T2/T3 波形 | `8/20 us` |
| 默认波形/线圈类型 | `8/20 us` |

## 事件生命周期

推荐状态机：

```text
idle
  -> armed
  -> triggered
  -> capturing
  -> finalized
  -> rearm_delay
  -> armed
```

说明：

- `armed`：高速 ADC 和预触发缓冲已就绪。
- `triggered`：检测到超过 raw 或积分前候选阈值的波形。
- `capturing`：继续采完整窗口，计算积分电流、峰值和时间特征。
- `finalized`：确认 `abs(peak_current) >= record_current` 后生成浪涌详情记录；如果同时超过 `alarm_current`，生成浪涌报警记录。
- `rearm_delay`：防止同一个波形被重复记录；多脉冲模式另行处理。

## 触发判定

最终产品判定条件分为两级：

```text
record_triggered = abs(peak_current_milli_ka) >= record_current_milli_ka
alarm_triggered  = abs(peak_current_milli_ka) >= alarm_current_milli_ka
```

其中：

- `peak_current_milli_ka` 来自高速 ADC 波形数字积分后的电流峰值。
- `record_current_milli_ka` 是浪涌详情记录准入阈值，典型值为 `500 / 1000 / 2500 / 5000`。
- `alarm_current_milli_ka` 是浪涌报警阈值，典型值为 `500 / 1000 / 2500 / 5000`。
- 未完成线圈参数标定前，不能把 raw ADC 峰值直接当作 kA 峰值。

工程调试阶段可临时使用 raw 阈值触发采样窗口，但记录中必须标记 `calibration_valid=0`，并且 `peak_current_milli_ka` 不得输出假换算值。

## 浪涌报警记录格式

浪涌报警记录用于报警页、蜂鸣器提示、上位机报警和报警历史。它只保存报警摘要，不重复保存完整浪涌波形和计算详情。

推荐结构：

```c
typedef struct
{
    uint32_t alarm_id;
    uint16_t format_version;
    uint16_t alarm_reason;             /* ALARM_REASON_SURGE_OVER_CURRENT */

    uint32_t timestamp_ms;
    uint8_t time_valid;
    app_rtc_datetime_t datetime;

    uint16_t alarm_current_milli_ka;    /* 本次比较使用的报警阈值 */
    int32_t peak_current_milli_ka;      /* 可选摘要，未标定时为 0 */
    uint16_t flags;

    uint32_t surge_sequence;           /* 对应浪涌详情记录 sequence */
    uint32_t surge_event_id;           /* 对应浪涌详情记录 event_id */
    uint32_t surge_tsdb_time;          /* 可选内部索引，未使用为 0 */
    uint32_t reserved;
} surge_alarm_record_v1_t;
```

报警原因建议固定为：

```text
ALARM_REASON_SURGE_OVER_CURRENT
含义：浪涌峰值大于等于 alarm_current_milli_ka
```

报警页显示时应能表达“具体大于多少阈值”，例如：

```text
浪涌峰值超过 5.0 kA
```

`surge_sequence` 和 `surge_event_id` 是稳定逻辑引用。后端可以优先从 RAM 最近记录缓存读取详情，缓存未命中时再由 record service 异步从 `surge_tsdb` 查询。不要把裸 Flash 物理地址作为对外协议字段；裸地址只能作为后端内部优化字段，避免后续分区、FlashDB 格式或擦写策略变化后导致引用失效。

## 完整 TSDB 记录字段

浪涌事件应写入 `surge_tsdb`。第一版建议使用独立结构承载完整字段，再由 `record_service` 统一分配 `sequence` 和通用头。

```c
typedef struct
{
    uint32_t sequence;
    uint16_t format_version;
    uint16_t record_type;              /* RECORD_SERVICE_EVENT_SURGE */

    uint32_t timestamp_ms;
    uint8_t time_valid;
    app_rtc_datetime_t datetime;

    uint32_t event_id;
    uint16_t record_current_milli_ka;   /* 500/1000/2500/5000 */
    uint16_t alarm_current_milli_ka;    /* 500/1000/2500/5000 */
    uint16_t waveform_type;            /* 1=8/20us, 2=10/350us */
    uint16_t sensor_profile_id;
    uint16_t flags;
    uint8_t alarm_triggered;
    uint8_t reserved0;
    uint32_t linked_alarm_id;           /* 未触发报警时为 0 */

    int32_t peak_current_milli_ka;      /* 带符号峰值，未标定时为 0 */
    uint16_t peak_raw;
    uint16_t baseline_raw;
    int16_t peak_delta_raw;
    uint16_t peak_sample_index;

    uint32_t front_time_ns;             /* 波前时间，未知为 0 */
    uint32_t half_value_time_ns;        /* 半峰值时间，未知为 0 */
    int32_t current_slew_milli_ka_per_us;
    int32_t charge_milli_coulomb;       /* Q 值，未知为 0 */
    int32_t energy_milli_joule;         /* 能量，未知为 0 */

    uint16_t sample_rate_khz;
    uint16_t capture_sample_count;
    uint16_t pretrigger_sample_count;
    uint16_t pulse_count;

    uint32_t multi_pulse_group_id;
    uint32_t previous_pulse_interval_us;
    uint16_t max_pulse_count_in_group;
    uint16_t reserved;
} surge_event_record_v1_t;
```

## 字段说明

| 字段 | 含义 |
| --- | --- |
| `sequence` | 全局历史记录序号，由 `record_service` 分配 |
| `format_version` | 浪涌事件记录格式版本，第一版建议为 `1` |
| `record_type` | 固定为 `RECORD_SERVICE_EVENT_SURGE` |
| `timestamp_ms` / `datetime` | 事件触发时间 |
| `event_id` | 浪涌事件 ID，可与 `sequence` 相同，也可由 `surge_service` 单独分配 |
| `record_current_milli_ka` | 记录准入阈值，单位 `0.001 kA` |
| `alarm_current_milli_ka` | 报警阈值，单位 `0.001 kA` |
| `waveform_type` | 线圈/波形类型：`1=8/20us`，`2=10/350us` |
| `sensor_profile_id` | 传感器参数组 ID，用于区分不同罗氏线圈或标定参数 |
| `flags` | 状态标志，见下表 |
| `alarm_triggered` | 本次浪涌是否触发高阈值报警，UI 可据此区分颜色 |
| `linked_alarm_id` | 对应浪涌报警记录 ID，未触发报警时为 0 |
| `peak_current_milli_ka` | 积分后的带符号峰值电流，单位 `0.001 kA` |
| `peak_raw` | 峰值点 ADC raw |
| `baseline_raw` | 触发前基线 ADC raw |
| `peak_delta_raw` | `peak_raw - baseline_raw`，带符号 |
| `peak_sample_index` | 峰值点在捕获窗口中的采样序号 |
| `front_time_ns` | 波前时间 |
| `half_value_time_ns` | 半峰值时间 |
| `current_slew_milli_ka_per_us` | 电流陡度 |
| `charge_milli_coulomb` | Q 值，电荷量 |
| `energy_milli_joule` | 估算能量 |
| `sample_rate_khz` | 本次捕获采样率 |
| `capture_sample_count` | 本次捕获窗口采样点数 |
| `pretrigger_sample_count` | 触发前采样点数 |
| `pulse_count` | 当前记录内识别到的脉冲数量，第一版可固定为 1 |
| `multi_pulse_group_id` | 多脉冲组 ID，第一版可为 0 |
| `previous_pulse_interval_us` | 与前一个脉冲的间隔 |
| `max_pulse_count_in_group` | 本组最多脉冲数，参考竞品可规划为 20 |

`flags` 建议定义：

| bit | 含义 |
| --- | --- |
| bit0 | `calibration_valid`：线圈和前端标定有效 |
| bit1 | `positive_polarity`：正极性 |
| bit2 | `negative_polarity`：负极性 |
| bit3 | `over_range`：ADC 或积分结果超量程 |
| bit4 | `raw_trigger_only`：仅 raw 触发，未形成有效 kA 换算 |
| bit5 | `multi_pulse_member`：属于多脉冲组 |
| bit6 | `front_time_valid` |
| bit7 | `half_value_time_valid` |
| bit8 | `charge_valid` |
| bit9 | `energy_valid` |
| bit10 | `alarm_triggered`：本次浪涌触发高阈值报警 |
| bit11 | `alarm_record_link_valid`：`linked_alarm_id` 有效 |

## 第一版必须保存的最小字段

如果高速 ADC 和积分算法还没完全确认，第一版至少应保存：

| 字段 | 原因 |
| --- | --- |
| `sequence` | 历史翻页主键 |
| `timestamp_ms` / `datetime` | 事件发生时间 |
| `record_current_milli_ka` | 记录当时的记录准入配置 |
| `alarm_current_milli_ka` | 记录当时的报警阈值配置 |
| `alarm_triggered` / `linked_alarm_id` | 支持 UI 区分是否报警，并从报警页跳到详情 |
| `waveform_type` | 区分 `8/20us` 和 `10/350us` |
| `flags` | 标记是否已标定、是否 raw-only、是否超量程 |
| `peak_raw` / `baseline_raw` / `peak_delta_raw` | 未标定阶段也能复盘波形强度 |
| `peak_current_milli_ka` | 标定有效后保存真实峰值 |
| `sample_rate_khz` / `capture_sample_count` | 方便追溯算法条件 |

波前时间、半峰值时间、Q 值、能量、电流陡度可以先预留，只有在采样率、积分算法和线圈参数确认后才置有效标志。

## Modbus 摘要映射建议

现有历史查询窗口 `Input 120..159` 中，`145..159` 只有 15 个寄存器可用于浪涌专属摘要，不适合塞入完整浪涌事件。建议第一版仅映射摘要：

| 地址 | 含义 |
| --- | --- |
| 145 | `surge_flags` |
| 146 | `record_current`，单位 `0.1 kA` |
| 147 | `peak_current`，单位 `0.1 kA`，按 int16 解释，未标定为 0 |
| 148 | `waveform_type << 8 | sensor_profile_id低8位` |
| 149 | `peak_raw` |
| 150 | `baseline_raw` |
| 151 | `peak_delta_raw`，按 int16 解释 |
| 152 | `front_time_us`，未知为 0，最大 65535us |
| 153 | `half_value_time_us`，未知为 0 |
| 154 | `sample_rate_khz` |
| 155 | `capture_sample_count` |
| 156 | `pulse_count` |
| 157 | `previous_pulse_interval_ms` |
| 158 | `charge` 摘要，单位待定，未知为 0 |
| 159 | `energy` 摘要，单位待定，未知为 0 |

`alarm_current`、`alarm_triggered`、`linked_alarm_id` 后续建议放入浪涌详情查询窗口，例如 `Input 200..239` 或文件/批量查询接口，不挤占通用历史翻页结果窗口。

## 与非浪涌报警的区别

| 项目 | 非浪涌报警 | 浪涌报警事件 |
| --- | --- | --- |
| 状态模型 | 有当前活跃报警 | 瞬时事件 |
| EEPROM 镜像 | 需要，保存未闭合报警 | 不需要 |
| TSDB 写入时机 | 报警发生写 `alarm_start`，报警恢复写 `alarm_clear` | 详情记录完成计算后立即写；超过高阈值时额外写 `alarm_instant` |
| 持续时间 | `alarm_clear` 事件中可带 `duration_ms` | 通常为 0 或捕获窗口时间 |
| 蜂鸣器 | 活跃期间断续报警 | 仅高阈值报警触发报警提示，低阈值记录不触发报警提示 |
| 查询方式 | 历史报警事件日志 | 浪涌详情记录；报警页通过逻辑引用查看详情 |

## 多脉冲规划

竞品资料提到可识别最多 20 个连续脉冲，并提到 `>=40 ms` 间隔。第一版不强制实现多脉冲详情，但格式先预留：

- `pulse_count`
- `multi_pulse_group_id`
- `previous_pulse_interval_us`
- `max_pulse_count_in_group`

建议先把单个脉冲事件跑稳定，再实现多脉冲组。多脉冲详情如果要记录每个脉冲的峰值、极性和间隔，需要独立的详情结构，不建议塞进通用记录头。

## 待确认问题

- 罗氏线圈互感、灵敏度和标定条件。
- 当前板级前端整体增益和频响。
- 高速 ADC 采样率、DMA 缓冲长度和触发前窗口。
- 数字积分算法和漂移校正策略。
- `8/20us` 和 `10/350us` 是否都必须支持。
- 能量计算是否有明确客户需求；仅靠电流波形无法严格得到 SPD 吸收能量，还需要系统电压或等效模型。
