# 参数体系设计

本文固定当前固件第一版参数体系。参数属于设备内部后台服务层，由 `param_service` 统一维护；UI、Modbus RTU、后续 Modbus TCP、MSH 等前端只能通过参数服务或 `backend_request` 修改参数，不直接写 EEPROM、KVDB 或参数结构体。

当前固件尚未对外发行，不保留开发期旧参数镜像兼容。参数表变动时优先保持 Modbus 地址稳定，不为兼容旧开发镜像增加迁移包袱。

## 设计目标

- 所有运行参数有统一 ID、名称、默认值、范围、单位和 Modbus Holding 地址。
- 参数读取只访问 RAM 参数影子，不能触发 EEPROM 或 Flash 慢操作。
- 参数保存到 AT24C128C EEPROM KVDB，历史日志不放 EEPROM。
- 前端写参数只提交请求，由 `backend_req` 串行处理，避免多个前端同时改参数。
- 参数修改后先进入 RAM 影子，只有执行保存命令后才持久化。
- 恢复默认参数只修改 RAM 影子，是否保存为默认值由后续保存命令决定。

## 当前架构

```text
Modbus / UI / MSH
  -> backend_request
  -> param_service
  -> storage_flashdb
  -> AT24C128C EEPROM KVDB
```

`param_service` 使用 RT-Thread mutex 保护参数表和状态。参数表在 RAM 中保存为 `param_service_values[]`，启动时先加载编译期默认值，再尝试从 EEPROM KVDB 读取 `param_image`。

EEPROM 当前使用整体镜像：

| 项目 | 当前值 |
| --- | --- |
| KV 名称 | `param_image` |
| magic | `0x5052414D` |
| storage version | `6` |
| 最大 entry 数 | `64` |
| entry 格式 | `{param_id, value}` |
| 校验 | 自定义 checksum |

加载时先装载默认参数，再按镜像 entry 中的参数 ID 覆盖对应值。未知 ID、越界值、版本不匹配或 checksum 错误都会导致本次镜像加载失败，并继续使用默认值。

## 参数状态

`param_service_status_t` 当前字段：

| 字段 | 含义 |
| --- | --- |
| `version` | RAM 参数表版本计数，参数加载或修改后递增 |
| `last_update_ms` | 最近一次参数状态更新时间 |
| `dirty` | RAM 参数是否与 EEPROM 持久化镜像不一致 |
| `persistent_ready` | EEPROM KVDB 参数区是否可用 |
| `count` | 当前参数数量 |

## 通信参数策略

Modbus RTU 参数固定为三项：

| 参数 | 含义 |
| --- | --- |
| `modbus_addr` | 从站地址 |
| `modbus_baudrate` | 波特率 |
| `modbus_data_format` | 串口数据格式，例如 `8N1` |

`modbus_data_format` 编码：

| 编码 | 格式 | 含义 |
| --- | --- | --- |
| `0` | `8N1` | 8 数据位，无校验，1 停止位 |
| `1` | `8E1` | 8 数据位，偶校验，1 停止位 |
| `2` | `8O1` | 8 数据位，奇校验，1 停止位 |
| `3` | `8N2` | 8 数据位，无校验，2 停止位 |
| `4` | `8E2` | 8 数据位，偶校验，2 停止位 |
| `5` | `8O2` | 8 数据位，奇校验，2 停止位 |

RTU 通信参数采用明确提交模型：

- 写 Holding `1..3` 或执行 `param set` 只修改 RAM 参数影子，不立刻改变正在运行的 `mb_rtu` 串口。
- 写 Holding `100=1` 或执行 `param save` 只保存 RAM 参数影子到 EEPROM KVDB，不立刻改变正在运行的 `mb_rtu` 串口。
- 写 Holding `100=5` 才把当前 RAM 参数影子应用到正在运行的 Modbus RTU 链路。
- 如果只写参数并保存，不执行 apply，则新通信参数在下次上电或复位时生效。

这样做是为了避免上位机写错波特率或数据格式时，单次 Holding 写入立即打断当前 RS485 链路。

## 网络 / Modbus TCP 参数策略

网络参数属于后台参数服务，由 EEPROM KVDB 持久化。第一版联网配置入口以 RS485 / Modbus RTU 为主，MSH 仅作为开发调试入口。虽然当前已经接入 W5500 Modbus TCP Server，但 Modbus TCP 不作为第一版推荐的网络参数配置入口，避免用户通过 TCP 修改自身 IP、网关或端口后导致当前连接失联。

IPv4 参数在 `param_service` 内部保存为一个 32 位值，对外 Modbus 使用 4 个连续 Holding，每个 Holding 保存一个字节，顺序与点分十进制一致。例如 `192.168.1.223` 对应 Holding 值 `192, 168, 1, 223`。

写 IPv4 参数必须一次写完整 4 个连续寄存器：

- IP 地址：一次写 Holding `51..54`。
- 子网掩码：一次写 Holding `55..58`。
- 网关：一次写 Holding `59..62`。

不允许只写单个 IP 字节，避免 RAM 或 EEPROM 中保存半个新 IP。写入只修改 RAM 参数影子；需要断电保持时执行保存命令 `100=1`；需要运行时应用时执行网络应用命令 `100=6`。当前 `100=6` 已接入 W5500 网络参数应用流程，会按 RAM 参数重新配置 W5500，并重新监听 Modbus TCP 端口。

Modbus TCP 第一版采用 W5500 硬件 TCP/IP 协议栈，固件只使用 `socket0` 作为 TCP server。因此第一版只支持 1 路 Modbus TCP client 并发连接。这个限制是当前产品策略，不是临时缺陷；如后续需要多上位机同时连接，再单独评估 W5500 socket 分配、RAM 占用、请求仲裁和异常断开恢复策略。

MAC 地址不放在 EEPROM 网络参数中。当前已经在片内 Flash 最后一个 2KB 扇区预留 `factory` 区，factory 区用于保存出厂身份信息，第一版保存固定 MAC 地址。`param defaults` 或恢复默认参数不会清除 factory MAC。

Modbus Holding `64..69` 用于读写 factory MAC。每个寄存器保存一个字节，顺序为 MAC 文本顺序，例如 `02:57:50:00:00:01` 对应 `2, 87, 80, 0, 0, 1`。写入时必须一次写完整 6 个连续寄存器，不能只写单个字节。Modbus 回调只投递后台请求，实际擦写 factory 区由 `backend_req` 完成；写入完成状态通过命令状态寄存器 `102/103` 查询。写入 factory MAC 后，正在运行的 W5500 不会自动切换 MAC；需要执行网络应用命令 `100=6` 后才会重新配置 W5500。

当前 MAC 地址策略分为两阶段：

- 开发阶段：factory 区未写入有效 MAC 时，固件使用本地管理单播 MAC 地址，前缀为 `02:57:50`，后 3 字节由当前 IP 派生。该策略只用于开发和小规模内部测试，可避免与公开 OUI 冲突，但不保证量产唯一性。
- 生产阶段：每台设备必须使用唯一 MAC 地址，由生产工具写入 factory 区并读回校验。可选来源包括客户分配 MAC 段、购买/申请 OUI，或使用本地管理 MAC 段并由生产数据库保证不重复。

量产 MAC 规则确认前，不把 fallback 的 `02:57:50:xx:xx:xx` 视为正式生产地址，也不承诺跨不同 IP 配置保持同一个 MAC。

## 报警参数策略

- TI1..TI5 报警默认关闭。
- NTC 传感器默认使能，但 NTC 高温报警和低温报警默认关闭。
- 浪涌报警默认开启。
- NTC 报警不设置独立恢复阈值或恢复回差；高温报警在温度低于高温阈值后恢复，低温报警在温度高于低温阈值后恢复。
- NTC 报警发生和恢复都需要状态持续超过 `ntc_alarm_debounce_ms` 后才切换。
- 浪涌详情参数等待浪涌详情方案单独收口。

## Modbus 映射规则

- Holding `1..99` 为运行参数区，读写 RAM 参数影子。
- Holding `64..69` 为 factory MAC 特殊区，读写片内 Flash factory 区，不进入 EEPROM 参数镜像。
- Holding `100..107` 为异步慢命令区。
- Modbus 回调只做范围校验和请求投递，不直接执行 EEPROM、Flash、ADC、I2C 等慢操作。
- 修改成功后 `dirty=1`，表示 RAM 参数尚未保存到 EEPROM。

特殊编码：

| 参数 | Modbus 编码 |
| --- | --- |
| `alarm_peak_milli_kv` | 寄存器单位为 0.1 kV，代码中写入时乘 100，读取时除 100 |
| 温度/湿度偏移 | 按 int16 解释，单位沿用内部参数单位 |
| NTC 高/低温报警阈值 | 寄存器单位为 0.01 摄氏度，代码中写入时乘 10 转为 milli_C，读取时除 10 |
| IPv4 地址 | 4 个连续 Holding，每个 Holding 一个字节 |
| factory MAC | 6 个连续 Holding，每个 Holding 一个字节 |
| mask / bool / 端口 / 地址 | 按无符号 16 位解释 |

## 持久化流程

启动加载：

```text
param_service_init()
  -> 加载编译期默认值到 RAM
  -> storage_flashdb_init()
  -> param_service_load()
  -> EEPROM 镜像合法时覆盖 RAM 参数
  -> EEPROM 不可用或镜像非法时继续使用默认值
```

修改参数：

```text
前端写参数
  -> backend_request_set_params_async()
  -> backend_req 线程
  -> param_service_set_i32()
  -> RAM 参数更新，version 递增，dirty=1
```

保存参数：

```text
前端写保存命令
  -> backend_req 线程
  -> param_service_save()
  -> 写 EEPROM KVDB param_image
  -> dirty=0，persistent_ready=1
```

加载参数：

```text
前端写加载命令
  -> backend_req 线程
  -> param_service_load()
  -> EEPROM 镜像合法时覆盖 RAM 参数
```

## 待确认问题

- 生产阶段 MAC 地址的来源、唯一性保证、SN/MAC 绑定台账和出厂工具流程待确认。
- MAC 地址已经开放 Holding `64..69` 和 MSH `factory mac set` 写入 factory 区；生产阶段是否需要写保护、授权或解锁机制待确认。
- NTC Beta、分压电阻和温度校准参数是否开放给用户配置待确认；当前硬件按 10k/10k/B3380 实现。
- 浪涌线圈和浪涌详情参数等待浪涌详情方案单独设计。

## 冻结原则

第一版参数表在进入上位机开发前应尽量稳定。冻结后：

- 不随意改变已有参数 ID。
- 不随意改变已有 Modbus Holding 地址。
- 不改变已有参数单位和缩放。
- 新增参数追加到预留区或表尾，除非明确决定不兼容开发期固件。
- 修改参数默认值必须同步更新本文档和 `Modbus临时寄存器表.md`。
